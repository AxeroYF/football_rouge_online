# 足球 Roguelite 比赛模拟器 v0.2

这是一个不依赖游戏引擎的 7 人制比赛模拟核心。它用事件链生成比赛，而不是直接根据综合评分抽取一个比分，因此球员、自由站位、系统识别阵型、战术、羁绊、体能、红牌、天气和比分状态都能通过具体比赛事件影响结果。

## v0.2 七人制统一架构

玩家侧 Demo、开发者后台和命令行批量模拟现在共用同一个七人制比赛模型与数据字典。统一规则固定为 7 名首发、最多 4 名替补、每场 3 个换人名额；位置只使用门将、中后卫、左/右边后卫、后腰、前腰、左/右中场、中锋和左/右边锋，不包含十一人制阵容模板。

球员统一使用 26 项能力、主位置、副位置、惯用脚和即时状态。旧存档与旧后台数据中的 `CM/WB/CF/W` 等位置会自动迁移到对应的七人制位置，`AM` 作为正式前腰位置保留，旧版精简能力也会展开为 26 项能力。共享比赛会话支持逐分钟推进、半场暂停、实时战术更新和玩家手动换人，因此开发者后台的模拟基础与玩家实际比赛不再是两套独立模型。

## 可玩 Demo

当前已加入七人制 Roguelite 玩家侧 Demo，包含自由拖动站位与自动阵型识别、赛前首发调整、对手阵型与球员能力预览、特性卡构筑、3 分钟事件驱动比赛、中场磁贴换人、进球演出、加时与点球、赛后重要时间轴，以及每场胜利两轮特性卡三选一。门将、后场、中场和前场必须各至少一人，但不再给玩家预设阵型；具体角色会按战术板横向与纵向位置识别，阵型结构和进攻策略会实际改变攻防、推进、机会与转换风险。每三场胜利会触发球员三选一，球员候选与商店球员盲盒均可出现 7 名传奇球星。商店每个候选位从 4% 起步、通关招募从 6% 起步，两条渠道分别在连续未出后逐轮提高，并于各自第 10 轮强制出现至少一名可用传奇；传奇带 3 张强力锁定特性并最多出战 10 场。

每名普通球员获取时固定携带 1 张随机先天特性，它占用槽位且不可卸下或替换。特性阵容页右侧和球队信息页提供“快捷出售”窗口，可把背包特性卡、可替换的后天卡或球员磁贴拖入窗口出售；首次确认可以勾选以后不再显示，也可以批量出售背包卡和替补球员。胜利金币默认采用 `220 + 当前关卡 × 20`，用于支撑更频繁的商店构筑。按 `F8` 会开启本地开发者后门：金币与商店直售刷新次数均显示为无限，当前及后续比赛结算必定获胜。

天气权重默认为晴天60%、雨天15%、雷暴15%、雪天10%。比赛伤病分为常规身体伤病和犯规踢伤；任何比赛内伤病都会立即暂停玩家侧比赛并强制伤员离场，有可用替补和换人名额时必须选择替补，否则确认少一人继续。伤情从至少伤停1场延伸到极低概率的生涯终结伤病；造成踢伤的犯规球员会显著提高吃黄牌和直接红牌的概率。雷暴仍按每次推进0.6%的默认概率触发雷击，使随机场上球员重伤离场并固定伤停5场，同时扣除体力和状态。伤病、判罚、换人、少人作战和赛后伤停均使用同一共享比赛会话。

    npm run demo

打开 http://127.0.0.1:4310/game/。Demo 使用浏览器本地存档；开发者后台仍位于 http://127.0.0.1:4310/。

## 11V11 好友对战原型

独立入口位于 http://127.0.0.1:4310/versus/，不会修改七人制征程逻辑。当前支持创建和加入六位房间码、双方在两分钟内完成 11 次真实球员三选一、为每名球员绑定随机先天特性与一张三选一特性，以及一分钟自由战术板准备。双方准备或倒计时结束后自动开始固定两分钟的十一人比赛；本模式没有替补与换人，红牌和伤病会直接造成少人作战。每方有一次最长30秒的临场战术暂停，平局支持加时和点球大战，赛中提供详细对抗播报、体能与实时评分，赛后生成双方完整报告。雷暴天气沿用原有权重，并保证本场至少出现一次雷击伤退。

默认服务仅允许本机访问。可信局域网联调时可使用 `VERSUS_HOST=0.0.0.0 npm run demo`，再把本机局域网地址和房间码发给好友；该原型暂不包含公网账号、断线续传或持久化房间。

## 开发者后台

项目包含独立的本地开发工具“场边实验室”，用于管理特性卡、球员卡和可视化比赛模拟参数。它仅监听本机地址，不进入正式游戏构建。

特性卡管理现分为“正式卡池”和“开发中”两个隔离工作区。开发中草稿允许缺少稀有度、位置、标签、羁绊与程序规则，也可以保存未完成的伪代码或 JSON；这些草稿存放在独立的 `traitDrafts` 集合，不参与玩家抽取、球员装备或比赛模拟。测试通过后的正式合并暂不自动执行，需要先共同评审设计与程序实现。

    npm run devtool

打开 http://127.0.0.1:4310/。开发数据保存在 data/devtool-db.json，每次保存前自动保留 data/devtool-db.backup.json。

## 当前模型

每次控球依次经过以下阶段：

1. 后场组织对前场逼抢
2. 中场推进对拦截与保护
3. 机会创造对整体防守
4. 射门类型、射手能力与门将扑救
5. 进球、扑救、偏出、角球或定位球

每个阶段使用同一种可解释概率结构：

    P(成功) = sigmoid(logit(基础概率) + 进攻优势 + 战术对位 + 比赛状态 + 环境影响)

比分不是固定预测值，而是一整个概率分布。批量运行数千场后，才能比较一套阵容或战术的真实期望。

## 参数层级

### 球员

每名球员目前有 26 项能力：

- 技术：传球、停球、盘带、传中、射门、远射、头球、定位球
- 防守：抢断、盯人、站位
- 思维：视野、决策、冷静、无球、纪律、侵略性
- 身体：速度、加速、力量、耐力、灵活、弹跳、工作投入
- 门将：守门、反应

即时状态包括体力、状态和受伤倾向。比赛位置细分为门将、中后卫、左/右边后卫、后腰、前腰、左/右中场、中锋、左/右边锋；主位置、副位置和惯用脚共同决定站位适配，错误位置会造成明显损失。有位置要求的特性卡只允许装备给主位置或副位置匹配的球员，全位置卡不受限制。

### 球队

- 战术：节奏、直接程度、宽度、逼抢、防线高度、冒险程度、抢断强度、反击、传中、定位球投入、拖延时间
- 阵型：防守平衡、中场密度、进攻人数、宽度、攻守转换风险、结构合理度
- 教练：进攻、防守、适应、换人倾向
- 团队状态：球员共同出场会积累两两默契，首发配对网络形成球队默契评分并提供比赛加成；另有全队状态与近期表现

### 比赛环境

- 主场优势
- 降雨、风、温度、场地质量
- 裁判严格程度、点球倾向、主场倾向
- 比赛时长、比赛节奏和控球序列数量

犯规、红黄牌、点球、伤病、换人、疲劳以及领先/落后的比赛策略会在比赛中动态改变这些指标。当前裁判参与度处于偏高测试档，便于观察判罚体验。

## 运行

电脑已经安装 Node.js，因此不需要下载依赖：

    npm test
    npm run simulate

也可以指定模拟数量和随机种子：

    node src/cli.js --matches 5000 --seed my-seed

相同种子会产生完全相同的单场比赛，便于复现问题和调参。

## 特性卡

当前卡池包含80张卡：普通30、稀有26、史诗17、传奇7。基础规则为随机三选一、按球员位置过滤、同一球员不重复、每人最多三个槽位，并设有稀有和史诗保底。

    npm run traits

初版评审稿位于 outputs/足球特性卡设计-v1.md，新增40张清单位于 outputs/足球特性卡设计-v2-80张.md。特殊比赛效果目前以规则钩子的形式定义，确认卡牌风格后再逐项接入事件模拟器。

## v0.2 当前校准结果

示例中的河湾竞技对铁城联运行 5,000 场：

- 主胜 54.24%，平局 23.32%，客胜 22.44%
- 平均比分期望 1.809 比 1.091
- 平均总进球 2.899
- 平均射门 15.839 比 10.616
- 最常见比分为 1-1、2-1、1-0 和 2-0

这个分布已达到“游戏原型可用”的量级，但还不是现实比赛预测器。当前系数由足球统计模型的通用结构和宽范围目标校准，没有使用某个真实联赛的数据拟合。

## 统计基础

足球进球数使用 Poisson 类模型作为校准锚点。Dixon 与 Coles 的经典模型在 Poisson 回归基础上处理球队实力的动态变化与低比分相关性：

https://www.research.lancs.ac.uk/portal/en/publications/modelling-association-football-scores-and-inefficiencies-in-the-football-betting-market%28d16276a2-d6e0-483b-a708-1d29663f1992%29.html

本原型没有直接抽取两个 Poisson 比分，而是使用动态事件强度；这是为了让游戏参数可以产生可见、可解释的比赛过程。关于红牌、落后状态和补时会改变进球强度的实证依据，可参考：

https://arxiv.org/abs/2312.04338

## 下一轮应做的校准

1. 持续校准 7 人制的控球数、射门数与进球基线。
2. 选定一个真实联赛数据集，拟合基础进球、射门、主场、红牌和比分状态系数。
3. 建立参数敏感性报告，保证每提高 5 点能力都产生稳定但不过分的变化。
4. 将 80 张特性卡的动态 JSON 规则逐类接入共享比赛会话。
5. 加入球员性格缺陷、长期伤病、停赛和 Roguelite 随机事件。
